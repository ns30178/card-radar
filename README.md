## 1. 核心目標與可行性評估

### 1.1 核心價值

台灣信用卡市場最大的痛點不是「資訊太少」，而是**資訊過載且高度碎片化**：同一張卡在不同通路、不同期間、不同登錄狀態下，回饋率天差地別；哩程卡又用一套完全不同的單位（幾元 1 哩）在計算，一般人根本無從比較。

本專案的核心價值可以濃縮成三句話：

- **統一度量衡**：把現金回饋、點數、哩程全部換算成「等值現金回饋率」，讓所有卡在同一個排行榜公平競爭。
- **情境化決策**：使用者不是找「最強的卡」，而是找「最適合我這種消費習慣的卡」。以 8 大實用情境切入，貼近真實生活。
- **零維護、免信任**：純靜態網站、無伺服器、無金流、無個資蒐集。資料自動更新，作者不需每天盯著，使用者也不必擔心被導購。

### 1.2 效益評估（市場面 + 技術面）

| 面向 | 評估 |
|---|---|
| 市場需求 | 高。「信用卡推薦」「海外刷卡回饋」等關鍵字長期高搜尋量，Money101、Mr.Market、卡優等內容站林立，代表需求真實存在。 |
| 差異化 | 中高。多數現有站是「人工撰寫的長文」，更新慢、難比較。本站主打**可運算的結構化資料 + 個人化計算機**，體驗差異明顯。 |
| 維運成本 | 極低。GitHub Pages（免費）+ GitHub Actions（免費額度內）+ 外部 cron 喚醒，幾乎零成本。 |
| 變現潛力 | 中。可走聯盟行銷（辦卡導流）或純作品集/工具定位。但要留意獨立性與利益揭露。 |
| 法遵風險 | 需注意。金融資訊有正確性與時效性風險，必須有明確免責聲明並強調「以官方為準」。 |

### 1.3 有辦法達到預期目標嗎？最大的技術痛點是什麼？

**結論：可以達成，但成敗完全取決於「資料正確性與新鮮度」這一關。** 技術架構本身不難（靜態網站是成熟方案），真正的挑戰在資料。

最大的三個技術痛點：

1. **回饋規則的非結構化與極端複雜（最大痛點）**
   銀行的活動辦法是給人看的行銷文案，不是給機器讀的資料。同一個「5% 回饋」背後可能藏著：需登錄、需綁定數位帳戶、需當月滿額、限指定通路、有每月上限、有活動到期日、新舊戶不同⋯⋯。要把這些「濃縮成機器可運算、又方便前端呈現的重點摘要」，是整個專案的靈魂，也是最難的地方。**本專案的解法是用 AI 做結構化抽取（見第 2、3 章），並輔以人工維護的保底資料庫確保底線品質。**

2. **資料來源的穩定性與時效性**
   PTT RSS 可能掛掉、格式可能改、也可能某段時間沒有優質情報。**解法：保底神卡資料庫墊檔（至少 40~50 張），畫面永不開天窗。**

3. **AI 服務的不穩定與流量限制**
   免費/低價的 AI API 常遇到 429（流量限制）與偶發性 5xx。**解法：智慧煞車（遇 429 自動冷卻 60 秒）+ 多模型備援（flash 掛了換 pro）+ 全程 try/except 不中斷主流程。**

> ⚠️ 誠實的風險提醒：AI 抽取的回饋數字**不可能 100% 正確**，銀行活動也隨時在變。因此本站的定位必須是「快速比較的參考工具」，而非「精準的權益保證」，並在 UI 上反覆強調「以官方公告為準」。這是產品定位問題，不是能不能做出來的問題。

## 2. 系統架構與資料庫欄位設計

### 2.1 整體架構（四層）

```
┌─────────────────────────────────────────────────────────────┐
│  外部排程 (cron-job.org)  ──每月喚醒──▶  GitHub Actions        │
└─────────────────────────────────────────────────────────────┘
        │ 觸發 python -m src.build
        ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 1 資料獲取層  │──▶│ 2 AI 處理大腦 │──▶│ 3 篩選/換算層 │──▶│ 4 前端展示層  │
│ ingest.py    │   │ ai_processor │   │ logic.py     │   │ index.html   │
│ PTT RSS +    │   │ 結構化抽取    │   │ 哩程換算/門檻 │   │ 讀靜態 JSON  │
│ 保底資料庫    │   │ 容錯/多模備援 │   │ 分類/排序     │   │ GitHub Pages │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
                                              │ 拋轉
                                              ▼  docs/data/*.json（靜態檔）
```

**技術選型建議：**

- **前端**：純 HTML + CSS + 原生 JS（單檔 `index.html`）。不需框架，載入極快，零建置。
- **資料處理**：Python（`feedparser` 解析 RSS、`google-generativeai` 呼叫 Gemini）。
- **排程/CI**：GitHub Actions（執行）+ cron-job.org（每月從外部喚醒，避免 GitHub 對長期閒置排程降頻）。
- **託管**：GitHub Pages（免費、無伺服器、CDN 快取）。
- **資料格式**：靜態 JSON。後端只在建置時跑一次，前端只讀檔，符合「零維護、全自動」目標。

### 2.2 資料庫欄位設計（本專案的關鍵）

沒有真正的資料庫，資料以 JSON「文件」形式存在（保底庫 `data/fallback_cards.json`、產出檔 `docs/data/*.json`）。核心是**一張卡一個物件**，欄位刻意拆成兩組：

- **機器可運算組**：給演算法排序、計算用（純數字/布林值）。
- **人類可閱讀組**：給前端直接顯示的重點摘要（AI 濃縮後的白話文）。

這正是回應「把極度複雜的回饋條件簡化提煉成機器可運算、又方便前端呈現的重點摘要」的設計。

| 欄位 | 型別 | 分組 | 說明 |
|---|---|---|---|
| `id` | string | — | 唯一鍵，用於去重（PTT 新資料蓋過保底資料） |
| `name` / `bank` / `network` | string | 顯示 | 卡名、發卡行、Visa/Master/JCB |
| `categories` | string[] | 運算 | 該卡歸屬的 8 大情境（一卡可多屬） |
| `reward_type` | enum | 運算 | `cash` / `miles` / `points` |
| `scenario_rates` | map | **運算** | 各情境的回饋率（%），排序與計算的主要依據 |
| `base_rate` | number | 運算 | 基礎（未加碼）回饋率 |
| `effective_rate` | number | **運算** | **等值現金回饋率**（哩程卡自動換算後填入），跨型態比較的統一尺標 |
| `mile_spec` | object | 運算 | 哩程卡專用：`ntd_per_mile`（幾元 1 哩）、`program`（航空計畫） |
| `cap` | object | **運算** | 上限：`amount`（金額，null=無上限）、`period`、`note` |
| `conditions` | object | **運算** | `need_register`、`need_digital_account`、`min_spend`、`mobile_pay_required` |
| `annual_fee` | object | 運算 | `amount`、`waivable`、`waive_condition` |
| `summary` | string | **顯示** | 一句話重點摘要（前端卡片標題用） |
| `highlights` | string[] | 顯示 | 賣點標籤 |
| `limitations` | string | **顯示** | AI 把複雜的上限/登錄/門檻濃縮成一段白話（點擊展開時顯示） |
| `valid_until` | date | 運算/顯示 | 活動到期日（可用於自動淘汰過期優惠） |
| `source` / `source_type` | string | — | 來源連結；`ptt` 或 `fallback` |

**「複雜規則 → 重點摘要」的提煉流程：**

```
銀行行銷長文 / PTT 情報文
        │  AI（ai_processor.py）依固定 JSON schema 抽取
        ▼
┌─────────────────────────────┬─────────────────────────────┐
│ 機器可運算欄位                │ 人類可讀欄位                  │
│ scenario_rates: {overseas:5} │ summary:「海外實體 5% 無上限」 │
│ cap: {amount:null}           │ limitations:「須登錄，限實體  │
│ conditions:{need_register:1} │  通路，非指定地區回落 1%」     │
└─────────────────────────────┴─────────────────────────────┘
        │
        ▼  同一物件同時餵給「排序演算法」與「前端卡片」
```

## 3. 篩選條件與推薦邏輯

### 3.1 必備且實用的進階篩選條件

以下條件都對應到資料庫欄位，皆可在前端做即時篩選（本骨架已內建部分，其餘為建議擴充）：

- **消費情境**（`categories`）：8 大類 —— 海外、日韓、歐美、哩程、網購、行動支付、國內無腦、外送影音。
- **回饋型態**（`reward_type`）：只看現金 / 接受點數 / 接受哩程。
- **免登錄**（`conditions.need_register = false`）：懶人取向，不想每月登錄。
- **無腦刷 / 無上限**（`cap.amount = null`）：不想被上限綁手綁腳。
- **免年費**（`annual_fee.amount = 0` 或 `waivable = true`）。
- **免綁數位帳戶**（`conditions.need_digital_account = false`）。
- **免消費門檻**（`conditions.min_spend = 0`）。
- **回饋率下限**：只看等值回饋率 ≥ X%。
- **活動未過期**（`valid_until >= today`）。

### 3.2 核心換算：讓哩程卡公平參賽

哩程卡最難比較，因為單位不同。本專案用**動態匯率**把它換算成等值現金回饋率：

```
等值現金回饋率(%) = (每哩現金價值 / 幾元1哩) × 100
```

- 預設每哩現金價值 = 0.5 元（可在 `config.MILE_VALUE_NTD` 調整）。
- 例：某卡「20 元 1 哩」 → (0.5 / 20) × 100 = **2.5%**。
- 例：某卡「10 元 1 哩」 → (0.5 / 10) × 100 = **5.0%**。

如此一來，哩程卡與現金回饋卡就能在同一張排行榜上直接比大小。

### 3.3 門檻過濾與分類排序

1. **正規化**：所有卡先跑 `normalize_card()`，哩程卡自動換算 `effective_rate`。
2. **去重**：以 `id` 去重，`source_type = ptt` 的新資料優先蓋過保底資料。
3. **門檻過濾**：只保留 `effective_rate > 1.0%` 的卡片，直接排除沒有競爭力的「廢卡」（門檻可調）。
4. **分類**：依 `categories` 把卡歸進 8 大情境。
5. **排序**：每個情境內，依「該情境回饋率」由高到低排序，取前 10 名，拆成 **主力（1–5）** 與 **備用（6–10）**。

### 3.4 最佳推薦邏輯（動態計算「哪張最適合」）— 選配後端 API

排行榜回答的是「這個情境誰最強」，但每個人的消費結構不同。`logic.recommend()` 提供一個可依消費輪廓算出「實拿最多」的演算法（前端目前改以「比較總表 + 篩選器」呈現，此函式保留為可選配的後端能力，未來要做問卷式推薦時可直接沿用）：

```
輸入：使用者每月各情境消費金額 spend_profile = {overseas: 20000, online_shopping: 10000, ...}

對每一張卡 c：
    月回饋 = Σ  min( spend[情境] × rate[情境] / 100 ,  該卡月上限 )
                （只加總這張卡有支援、且使用者有花錢的情境）
    年回饋 = 月回饋 × 12

依「月回饋」由高到低排序 → 第 1 名就是「最適合你、實拿最多」的卡
```

關鍵細節：**計算實拿回饋時要套用上限（cap）**。一張帳面 11% 但每月上限 300 元的卡，對高消費者的實際貢獻可能遠不如一張 3.5% 無上限的卡。這正是「神卡不是看最高 %，而是看實拿」的量化實現。

> 進階可擴充：加入「多卡組合最佳解」（每個情境各挑最強的卡，算出理論極大值），或把年費、哩程兌換效率一起納入淨回饋計算。

## 4. UI/UX 介面呈現與分頁設計

### 4.1 動線與分頁架構

採「單頁、情境分頁（Tab）」設計，動線最短：

```
首頁（index.html）
├── 情境分頁列（8 個情境 Tab + 1 個「📊 比較總表」Tab）
│     🌍 海外通用 ｜ 🗾 日韓 ｜ 🗽 歐美 ｜ ✈️ 哩程
│     🛒 網購 ｜ 📱 行動支付 ｜ 🏠 國內無腦 ｜ 🍔 外送影音 ｜ 📊 比較總表
├── 實用篩選器（全站通用開關）：免登錄 ｜ 無上限 ｜ 免年費 ｜ 免數位帳戶
├── 情境分頁內：
│     👑 主力推薦（金色標記） / 🔥 備用神卡（灰色標記）
├── 比較總表分頁：所有卡並排的可排序表格（點欄位標題排序回饋率/上限/年費）
└── 頁尾：更新時間 + 免責聲明
```

明確區分不同用途，讓使用者「知道自己要哪一種」就能直達，不必讀落落長的文章。

### 4.2 直觀比較介面設計構想

- **一眼看懂的回饋數字**：每張卡右上角用大字綠色顯示「該情境等值回饋率 %」，排行榜由高到低，優劣一目了然。
- **主力 / 備用視覺分層**：主力卡金色皇冠 👑、備用卡火焰 🔥 + 灰階，強弱層次分明。
- **點擊展開詳細限制**：卡片預設只顯示「卡名 + 回饋率 + 一句話摘要 + 賣點標籤」，保持清爽；點擊後平滑展開，顯示 AI 濃縮的完整限制（上限、是否需登錄、是否需數位帳戶、消費門檻、年費、活動到期、重點限制）。這讓「快速掃描」與「深入研究」兩種需求並存。
- **全卡回饋比較總表**：獨立的「📊 比較總表」分頁，把所有卡並排成一張表，可點欄位標題排序（等值回饋率、上限、年費），最像專業金融比較平台。
- **實用篩選器**：全站通用的開關（免登錄／無上限／免年費／免數位帳戶），一鍵過濾掉條件麻煩的卡，同時作用於卡片檢視與比較總表。
- **金融質感視覺**：深色底 + 卡片懸浮陰影 + 金/綠點綴，營造專業比較平台的信任感。
- **RWD 響應式**：`grid` 自動換行，手機、平板、桌機皆適用。

（以上皆已實作於 `docs/index.html`，可直接開啟預覽。）

### 6.1 專案結構

```
credit-card-compare/

├── requirements.txt          # Python 依賴
├── config.py                 # 全域設定（分類/匯率/門檻/AI 模型/來源）

│   └── fallback_cards.json   # 硬體級保底神卡資料庫（48 張）
├── src/
│   ├── ingest.py             # 1 資料獲取層：PTT RSS + 保底
│   ├── ai_processor.py       # 2 AI 處理大腦：結構化抽取 + 容錯/多模備援
│   ├── logic.py              # 3 篩選/換算層：哩程換算/門檻/分類/排序/推薦
│   └── build.py              # 主流程：串起全部並拋轉靜態 JSON
├── docs/                     # ← GitHub Pages 根目錄
│   ├── index.html            # 4 前端展示層（單檔，含計算機）
│   └── data/                 # build 產出的靜態 JSON（含範例）
│       ├── index.json
│       ├── all_cards.json
│       └── <category>_top10.json  ×8
└── .github/workflows/
    └── build.yml             # GitHub Actions（cron + 外部喚醒 + 手動）
```

### 6.2 本機執行

```bash
pip install -r requirements.txt
# 若要啟用 PTT + AI 抽取，設定金鑰（沒設也能跑，僅用保底庫）
export GEMINI_API_KEY="你的金鑰"
python -m src.build          # 產出 docs/data/*.json
python -m http.server -d docs 8000   # 開 http://localhost:8000 預覽
```

> 未安裝 `feedparser` 或未設 `GEMINI_API_KEY` 時，系統會自動略過 PTT/AI，只用 48 張保底神卡建置，畫面依然完整（已驗證）。

### 6.3 自動化部署

1. 推上 GitHub，Settings → Pages → 來源選 `docs/` 資料夾。
2. Settings → Secrets → 新增 `GEMINI_API_KEY`。
3. `.github/workflows/build.yml` 已設定：每月 schedule + 手動 + `repository_dispatch`。
4. 到 cron-job.org 建立每月排程，呼叫 GitHub 的 `repository_dispatch` API（帶 `event_type: rebuild`）即可從外部喚醒建置。
